Overview

Introduction

Dev Hub is the shared developer area. It contains Markdown (.md) files that describe everything that supports development work but is not part of the product/API documentation itself. Dev Hub is for "how we work" and "how we support development". Do not place product design/architecture, API docs, or project feature documentation here. Those belong under the normal Doxygen sections (Project / Dependencies / Modules / Namespaces / Classes / Files, etc.).

Note
  • Doxygen does not infer page hierarchy from the folder structure. The Dev Hub tree must be built explicitly using \subpage links between pages.
  • Because of Tree Node Restriction, a node cannot have both a link and child items, this means using special _contents.md pages for a branch node (see Dev Hub Tree Structure below).
Warning
The entire Dev Hub branch depends on the main _contents.md file with Page ID dp_dh_dir_dev_hub to be present. This file is the starting point to get all other Markdown files in the Dev Hub branch. If this file is not found then Dev Hub branch appears only if Dev Hub Required Page is present and only that page will be inside Dev Hub branch.

Rules & Guidelines

  1. All files and folders should be inside the Dev_Hub/ folder.
  2. Only Markdown (.md) files are allowed inside Dev_Hub/ and its subfolders.
  3. For slugs (folder names, file basenames, page ids): only a-z0-9_ is allowed.
  4. Every Dev Hub Page ID must start with dp_dh_ (example: dp_dh_doxy_config).
  5. Every folder must contain a file named _contents.md.
    • This file defines the folder structure that will be visible in final output.
    • Its Page ID must start with dp_dh_dir_ (example: dp_dh_dir_dev_hub).
    • The title of this page (what appears after id) will be the folder name.
    • For details, see Dev Hub Tree Structure section below.
  6. Section ID should start with Page ID followed by two underscores.
    • For a page with Page ID \page dp_dh_some_page, make sections like this:
      # Section A {#dp_dh_some_page__section_a} where Section A is section title
      and dp_dh_some_page__section_a is section ID.
  7. Preferably, sections with child sections should not contain any other text.
    • Because of Tree Node Restriction a section with child sections will have its
      hash link discarded from the hash tree generated using doxy-plus.js file.
    • Clicking on a section with child sections only expands/collapses the children branch.

Tree Node Restriction

The structure of the nodes in the navigation trees (both main and hash navigation trees) generated using doxy-plus.js allows either a link or a list of child nodes, but not both. If a node has both, the list of child nodes takes priority and the link is discarded. This design choice is deliberate and it gives each node a single responsibility: either expand/collapse or navigate on click. As a result, the user interface is simpler and more intuitive.

Dev Hub Tree Structure

The tree structure for Markdown (.md) files are not automatically deduced from the folder structure. The tree structure is specifically deduced using \subpage links between pages. Because of Tree Node Restriction, any Markdown page containing \subpage ids will have the link to its own page contents removed from navigation tree and will be turned into a branch with the pages provided by the \subpage ids as its children. Due to this the preferred method is to have a special _contents.md file in each folder. This file's \page id should start with dp_dh_dir_ and the branch name is the title that appears after this id. This file should only contain the following:

  • Page ID using \page marker.
  • Title i.e. the text that appears after the Page ID.
  • Subpage IDs using \subpage marker.

Below is an example _contents.md page:

\page dp_dh_dir_settings Settings
\subpage dp_dh_basic_settings
\subpage dp_dh_expert_settings
\subpage dp_dh_dir_misc_settings

Here Settings becomes the branch name and pages with IDs dp_dh_basic_settings, dp_dh_expert_settings becomes child pages and page with ID dp_dh_dir_misc_settings (because it starts with dp_dh_dir_) becomes a sub-branch of the Settings branch.

Note
Each sub page provided by \subpage marker should have its own dedicated Markdown page where the same Page ID is assigned to \page. The title of the sub page is taken from the title provided in its own Markdown page just after the Page ID.
Warning
A markdown page with no title will be discarded. So, this \page dp_dh_somepage Some Page is valid while \page dp_dh_somepage is invalid.

Dev Hub Required Page

This page is automatically generated from documentation content tagged as required. Its purpose is to collect all such entries in one place, making important requirements easier to review, track, and maintain. It only appears when there is at least one usage of the required tag in any file. When it appears, it is placed directly below Overview page inside the Dev Hub branch.

This required page is for Dev Hub branch only, for project related required page have a look at Project Required Page.

Precondition
To enable this feature make sure that the following line is added in the Doxyfile:
ALIASES += "dh_req=\xrefitem dp_dh_required \"REQUIRED\" \"Required\""
Use += instead of = in Doxyfile when defining aliases so that you do not override other aliases.

To tag a line as required use:

\dh_req Some brief text that will appear at the location where it is written and also in the Required page.

To link to a specific section or page, include a reference in the text:

\dh_req See section \ref section_id
\dh_req See page \ref page_id

Insert Images In Markdown Files

Images can be inserted in .md pages.

  1. Create an _images/ folder at the root.
  2. In Doxyfile, add:
    • IMAGE_PATH = ./_images
    • (Multiple paths can be added)
  3. Copy the image file into _images/ (example: _images/test_img.png).
  4. In any .md file, reference it with either:
    • ![Image caption](test_img.png) (uses IMAGE_PATH)
    • ![Image caption](_images/test_img.png) (explicit path)
Warning
Keep image basenames globally unique across all IMAGE_PATH folders otherwise collisions can occur.
