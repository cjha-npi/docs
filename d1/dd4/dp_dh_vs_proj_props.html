Project Properties

Introduction

This document describes the recommended project properties for MSBuild-based C/C++ projects in Visual Studio 2022/2026.

These settings are intended to:

  • Keep build artifacts out of the repository/solution tree.
  • Ensure consistent language standards.
  • Enable Qt log context metadata in Release builds.
  • Provide a predictable include layout for project-owned headers.

The last section Automatic Assignment provides a way to have these properties assigned automatically for each new project.

Note
See Settings page for settings related to Visual Studio 2022/2026

Properties to Change

Output Directory

Project Properties > Configuration Properties > General > Output Directory

F:\Misc\VS_Bins\$(SolutionName)\$(Platform)_$(Configuration)\

This directory contains the final build outputs (for example: .exe, .dll, .pdb, .lib). It is recommended that projects inside a single solution share the same output directory, as long as output file names do not collide, to facilitate interaction between projects. Default is $(SolutionDir)$(Platform)\$(Configuration)\ (e.g. ...\MySolution\x64\Debug\).

Note
  • Safe to delete, Visual Studio/MSBuild recreates it when needed.
  • In .props xml file, it is refferred as OutDir.
  • If changed, the previously generated folder is not removed automatically.

Intermediate Directory

Project Properties > Configuration Properties > General > Intermediate Directory

F:\Misc\VS_Objs\$(SolutionName)\$(Platform)_$(Configuration)__$(ProjectName)\

This directory contains intermediate build artifacts (for example: .obj, .pch, moc_*.cpp, ui_*.h, qrc_*.cpp, generated .tlog, etc.). It is highly recommended that projects have different intermediate directory, even inside the same solution, to avoid name collisions. Default is $(ProjectDir)$(Platform)\$(Configuration)\ (e.g. ...\MyProject\x64\Debug\).

Warning
Projects should have their own intermediate directory, even inside the same solution, to avoid the possibility of name collision between build artifacts.
Note
  • Safe to delete, Visual Studio/MSBuild recreates it when needed.
  • In .props xml file, it is refferred as IntDir.
  • If changed, the previously generated folder is not removed automatically.

C++ Language Standard

Project Properties > Configuration Properties > General > C++ Language Standard Project Properties > Configuration Properties > C/C++ > Language > C++ Language Standard

ISO C++20 Standard (/std:c++20)
Note
The same value is found in both locations, changing one is sufficient, the change is reflected in other location as well.

C Language Standard

Project Properties > Configuration Properties > General > C Language Standard Project Properties > Configuration Properties > C/C++ > Language > C Language Standard

ISO C17 (2018) Standard (/std:c17)
Note
The same value is found in both locations, changing one is sufficient, the change is reflected in other location as well.

Additional Include Directories

Project Properties > Configuration Properties > C/C++ > General > Additional Include Directories

$(ProjectDir)inc;%(AdditionalIncludeDirectories)
  • $(ProjectDir)inc : Adds project-root\inc\ to include path.
    • Purpose:
      • Establish a consistent include root for project-owned headers.
      • Encourage includes with a prefix (example: #include "npi/bla.h") to reduce ambiguity.
    • Recommended project include layout:
      • project-root\inc\ -> added to "Additional Include Directories"
      • project-root\inc\npi\ -> headers native to npi
      • project-root\inc\ext\ -> external/vendor headers (only if you intentionally mirror them here)
    • With project-root\inc on the include path, project header usage becomes explicit:
      • #include "npi/bla.h"
      • #include "ext/some_lib/header.h"
        Note
        If a project does not have an inc folder, nothing fails just because the directory is missing. You only get an error if you actually include a header that would have been found only in that folder.

Preprocessor Definitions

Project Properties > Configuration Properties > C/C++ > Preprocessor > Preprocessor Definitions

QT_MESSAGELOGCONTEXT;%(PreprocessorDefinitions)
  • QT_MESSAGELOGCONTEXT : Qt’s log context fields (category, file, function, line) are not available in Release build, with this macro these are enabled in Release builds as well.

Subsystem

Project Properties > Configuration Properties > Linker > System > Subsystem

Windows (/SUBSYSTEM:WINDOWS)

GUI applications should use Windows subsystem; console tools should use Console subsystem. Some templates (including Qt “Empty Project”) may default to Console subsystem.

Note
A global .props imported early can be overridden later by the project file or other imported props/targets (templates often hardcode Console in the project). If subsystem does not change as expected, set the subsystem explicitly in that project properties or search the .vcxproj for <SubSystem> and remove/override the later setting.

Automatic Assignment

Procedure

Visual Studio C++ projects can auto-import a per-user property sheet if it exists:

  • LOCALAPPDATA%\Microsoft\MSBuild\v4.0\Microsoft.Cpp.x64.user.props (applies when building Platform = x64)
  • LOCALAPPDATA%\Microsoft\MSBuild\v4.0\Microsoft.Cpp.Win32.user.props (applies when building Platform = Win32)
Note
  • This is per-user (i.e. user acoount based) and per-platform (i.e. x64 and Win32);
  • The LOCALAPPDATA%\Microsoft\MSBuild\... directory may not exist by default. Create the directory and the files as needed.
Warning
The subsystem property assigned by this these .props files can be overridden later by the project file or other imported props/targets (templates often hardcode Console in the project). If subsystem does not change as expected, set the subsystem explicitly in that project properties or search the .vcxproj for <SubSystem> and remove/override the later setting.

Safeguard & Scope

These user props apply to any .vcxproj loaded/built via MSBuild under this Windows user account (Visual Studio builds, command line msbuild, CI jobs, scripts, etc.).

To avoid affecting unrelated projects, the provided code includes a safeguard so the defaults are applied only when the project is located under approved roots on F:\ (any depth):

  • F:\Test\...
  • F:\Eval\...
  • F:\Demo\...
  • F:\Core\...
  • F:\Uniq\...
  • F:\Gear\...

Code

Copy/paste the same XML into both files:

  • LOCALAPPDATA%\Microsoft\MSBuild\v4.0\Microsoft.Cpp.x64.user.props
  • LOCALAPPDATA%\Microsoft\MSBuild\v4.0\Microsoft.Cpp.Win32.user.props

(Use identical content unless you intentionally need platform-specific differences.)

<?xml version="1.0" encoding="utf-8"?>
<Project>
<!--
Microsoft.Cpp.<Platform>.user.props (per-user, per-platform)
This file is automatically imported (if it exists) by typical Visual Studio
C++ MSBuild projects (.vcxproj) when building the matching Platform:
- Microsoft.Cpp.x64.user.props -> applies when $(Platform) == x64
- Microsoft.Cpp.Win32.user.props -> applies when $(Platform) == Win32
Important behavior:
- This affects any build that loads/builds a .vcxproj via MSBuild
(Visual Studio IDE build, msbuild.exe, CI jobs, scripts, etc.).
- It does NOT affect build systems that do not use .vcxproj/MSBuild
(e.g., Ninja, Make, custom direct cl.exe invocations).
- It applies to both Debug and Release (and any other configurations)
unless you add additional conditions on $(Configuration).
Goal here:
- Apply a consistent OutDir/IntDir + language standards + a couple of
compile/link defaults
- ONLY for projects whose .vcxproj is located under specific "allowed"
roots on the F: drive:
F:\Test\...
F:\Eval\...
F:\Demo\...
F:\Core\...
F:\Uniq\...
F:\Gear\...
Notes on MSBuild concepts used below:
- PropertyGroup:
Defines MSBuild properties like OutDir and IntDir.
- ItemDefinitionGroup:
Defines default metadata for item types like ClCompile and Link.
This is the correct place for compiler/linker settings.
-->
<PropertyGroup>
<!--
$(MSBuildProjectDirectory) is the directory that contains the .vcxproj.
We append a trailing backslash to make prefix matching unambiguous and
robust. Example:
$(MSBuildProjectDirectory) = F:\Eval\MySol\MyProj
_ProjDirWithSlash = F:\Eval\MySol\MyProj\
-->
<_ProjDirWithSlash>$(MSBuildProjectDirectory)\</_ProjDirWithSlash>
<!--
_ApplyUserProps is our single "gate" switch.
It becomes "true" only if the project directory begins with one of our
approved roots. The trailing backslash in each root matters:
- F:\Test\ matches F:\Test\Anything\...
- F:\Test2\ does NOT match (avoids accidental partial matches)
OrdinalIgnoreCase ensures case-insensitive matching (F:\eval\... also matches).
-->
<_ApplyUserProps Condition="
$([System.String]::Copy('$(_ProjDirWithSlash)').StartsWith('F:\Test\', System.StringComparison.OrdinalIgnoreCase))
OR $([System.String]::Copy('$(_ProjDirWithSlash)').StartsWith('F:\Eval\', System.StringComparison.OrdinalIgnoreCase))
OR $([System.String]::Copy('$(_ProjDirWithSlash)').StartsWith('F:\Demo\', System.StringComparison.OrdinalIgnoreCase))
OR $([System.String]::Copy('$(_ProjDirWithSlash)').StartsWith('F:\Core\', System.StringComparison.OrdinalIgnoreCase))
OR $([System.String]::Copy('$(_ProjDirWithSlash)').StartsWith('F:\Uniq\', System.StringComparison.OrdinalIgnoreCase))
OR $([System.String]::Copy('$(_ProjDirWithSlash)').StartsWith('F:\Gear\', System.StringComparison.OrdinalIgnoreCase))
">true</_ApplyUserProps>
<!--
$(SolutionName) is often set when building via a .sln.
However, if you build a .vcxproj directly (or from some tooling),
$(SolutionName) can be empty. To avoid generating a path with a missing
segment, we create a stable fallback name.
This is only computed when _ApplyUserProps is true, because otherwise
we are not going to use it.
-->
<_SolName Condition="'$(_ApplyUserProps)'=='true' AND '$(SolutionName)' != ''">$(SolutionName)</_SolName>
<_SolName Condition="'$(_ApplyUserProps)'=='true' AND '$(SolutionName)' == ''">NoSolution</_SolName>
</PropertyGroup>
<!--
Build output/intermediate directories.
These are MSBuild properties (not compiler/linker flags), so they belong
in a PropertyGroup.
Important:
- Both OutDir and IntDir should end with a trailing backslash.
- These will apply to all configurations (Debug/Release/...) for the
current platform (x64 or Win32), as long as _ApplyUserProps is true.
Rationale:
- Prevents creation of local per-project folders like ".\x64\Debug\..."
(as long as this is in place BEFORE the first build of a project).
- Centralizes build artifacts so solution trees stay clean.
-->
<PropertyGroup Condition="'$(_ApplyUserProps)'=='true'">
<OutDir>F:\Misc\VS_Bins\$(_SolName)\$(Platform)_$(Configuration)\</OutDir>
<IntDir>F:\Misc\VS_Objs\$(_SolName)\$(Platform)_$(Configuration)__$(ProjectName)\</IntDir>
</PropertyGroup>
<!--
Default compiler and linker settings.
These are item metadata defaults, so they belong under ItemDefinitionGroup.
ClCompile:
- LanguageStandard: selects /std:c++20
- LanguageStandard_C: selects /std:c17
- AdditionalIncludeDirectories:
Adds "$(ProjectDir)inc" first, then keeps whatever the project
already had via %(AdditionalIncludeDirectories).
If "inc" does not exist, nothing fails by itself; it only matters
if you try to include headers from there.
- PreprocessorDefinitions:
Adds QT_MESSAGELOGCONTEXT, then preserves existing defines via
%(PreprocessorDefinitions).
Link:
- SubSystem: sets /SUBSYSTEM:WINDOWS by default.
You can override per-project to CONSOLE in that project's properties
(project settings normally take precedence over these defaults).
-->
<ItemDefinitionGroup Condition="'$(_ApplyUserProps)'=='true'">
<ClCompile>
<LanguageStandard>stdcpp20</LanguageStandard>
<LanguageStandard_C>stdc17</LanguageStandard_C>
<AdditionalIncludeDirectories>$(ProjectDir)inc;%(AdditionalIncludeDirectories)</AdditionalIncludeDirectories>
<PreprocessorDefinitions>QT_MESSAGELOGCONTEXT;%(PreprocessorDefinitions)</PreprocessorDefinitions>
</ClCompile>
<Link>
<SubSystem>Windows</SubSystem>
</Link>
</ItemDefinitionGroup>
<!--
If you ever want to restrict to only Debug or only Release, add to the
Condition, for example:
Condition="'$(_ApplyUserProps)'=='true' AND '$(Configuration)'=='Debug'"
If you ever want to apply different values for x64 vs Win32, you can:
- keep two files (x64 and Win32) with different values, OR
- branch on $(Platform) inside the same file (not recommended here
because you are duplicating the same file anyway).
-->
</Project>
