Dependencies Branch

Introduction

The Dependencies branch contains documentation for all project dependencies that are not part of the core project codebase and are therefore not documented through the project source files alone (.h, .cpp, etc.). This includes external tools, helper executables, additional source packages, runtime assets, generated files, and other required supporting components. Its purpose is to provide a clear reference for what each dependency is, where it is used, why it exists, and what is required to integrate or maintain it correctly.

The Dependencies branch is similar to Dev Hub (see Dev Hub Overview page) branch in the sense that both uses Markdown files, similar tree structure, similar rules and guidelines, and usage of Doxygen xrefitem for Required pages. The main difference between Dependencies and Dev Hub is that Dependencies are project dependent while Dev Hub is not.

Note
  • Doxygen does not infer page hierarchy from the folder structure. The Dependencies tree must be built explicitly using \subpage links between pages.
  • Because of Tree Node Restriction, a node cannot have both a link and child items, this means using special _contents.md pages for a branch node (see Dependencies Tree Structure below).
Warning
The entire Dependencies branch depends on the main _contents.md file with Page ID dp_dep_dir_dependencies to be present. This file is the starting point to get all other Markdown files in the Dependencies branch. If this file is not found then Dependencies branch appears only if Project Required Page is present and only that page will be inside Dependencies branch.

Rules & Guidelines

  1. All files and folders should be inside the _dependencies/ folder at the root.
  2. Only Markdown (.md) files are allowed inside _dependencies/ and its subfolders.
  3. For slugs (folder names, file basenames, page ids): only a-z0-9_ is allowed.
  4. Every Dependency Page ID must start with dp_dep_ (example: dp_dep_some_page).
  5. Every folder must contain a file named _contents.md.
    • This file defines the folder structure that will be visible in final output.
    • Its Page ID must start with dp_dep_dir_ (example: dp_dep_dir_misc).
    • The title of this page (what appears after id) will be the folder name.
    • For details, see Dependencies Tree Structure section below.
  6. Section ID should start with Page ID followed by two underscores.
    • For a page with Page ID \page dp_dep_some_page, make sections like this:
      # Section A {#dp_dep_some_page__section_a} where Section A is section title
      and dp_dep_some_page__section_a is section ID.
  7. Preferably, sections with child sections should not contain any other text.
    • Because of Tree Node Restriction a section with child sections will have its
      hash link discarded from the hash tree generated using doxy-plus.js file.
    • Clicking on a section with child sections only expands/collapses the children branch.

Dependencies Tree Structure

The tree structure for Markdown (.md) files are not automatically deduced from the folder structure. The tree structure is specifically deduced using \subpage links between pages. Because of Tree Node Restriction, any Markdown page containing \subpage ids will have the link to its own page contents removed from navigation tree and will be turned into a branch with the pages provided by the \subpage ids as its children. Due to this the preferred method is to have a special _contents.md file in each folder. This file's \page id should start with dp_dep_dir_ and the branch name is the title that appears after this id. This file should only contain the following:

  • Page ID using \page marker.
  • Title i.e. the text that appears after the Page ID.
  • Subpage IDs using \subpage marker.

Below is an example _contents.md page:

\page dp_dep_dir_ext_tools External Tools
\subpage dp_dep_basic_tools
\subpage dp_dep_expert_tools
\subpage dp_dep_dir_misc_tools

Here External Tools becomes the branch name and pages with IDs dp_dep_basic_tools, dp_dep_expert_tools becomes child pages and page with ID dp_dep_dir_misc_tools (because it starts with dp_dep_dir_) becomes a sub-branch of the External Tools branch.

Note
Each sub page provided by \subpage marker should have its own dedicated Markdown page where the same Page ID is assigned to \page. The title of the sub page is taken from the title provided in its own Markdown page just after the Page ID.
Warning
A markdown page with no title will be discarded. So, this \page dp_dep_somepage Some Page is valid while \page dp_dep_somepage is invalid.

Project Required Page

This page is automatically generated from documentation content tagged as required. Its purpose is to collect all such entries in one place, making important requirements easier to review, track, and maintain. It only appears when there is at least one usage of the required tag in any file. When it appears, it is placed under the Dependencies branch (if Dependencies branch does not exist it is created automatically).

This required page is for Project only, for Dev Hub related required page have a look at Dev Hub Required Page.

Precondition
To enable this feature make sure that the following line is added in the Doxyfile:
ALIASES += "dep_req=\xrefitem dp_dep_required \"REQUIRED\" \"Required\""
Use += instead of = in Doxyfile when defining aliases so that you do not override other aliases.

To tag a line as required use:

\dep_req Some brief text that will appear at the location where it is written and also in the Required page.

To link to a specific section or page, include a reference in the text:

\dep_req See section \ref section_id
\dep_req See page \ref page_id

Insert Images In Markdown Files

Images can be inserted in .md pages.

  1. Create an _images/ folder at the root.
  2. In Doxyfile, add:
    • IMAGE_PATH = ./_images
    • (Multiple paths can be added)
  3. Copy the image file into _images/ (example: _images/test_img.png).
  4. In any .md file, reference it with either:
    • ![Image caption](test_img.png) (uses IMAGE_PATH)
    • ![Image caption](_images/test_img.png) (explicit path)
Warning
Keep image basenames globally unique across all IMAGE_PATH folders otherwise collisions can occur.
