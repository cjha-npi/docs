Imp Log Process

Introduction

ImpLogProcess is a small helper executable that runs alongside the main application and exists solely to persist the most important log messages, such as cortical and fatal events, into a separate file through an independent process. Its purpose is to preserve critical diagnostics even when the host application is unstable, shutting down unexpectedly, or has already crashed and can no longer be trusted to flush its normal log pipeline. Because this process is isolated, minimal, and based only on direct Win32 pipe and file operations, it serves as a last-resort log sink whose reliability is intentionally decoupled from the health of the main executable.

Details

ImpLogProcess is intentionally narrow in scope. It does not decide which log messages are important, it does not interpret log contents, and it does not perform any formatting, filtering, timestamping, or severity classification. All of that remains the responsibility of the host application. The helper process only receives a raw byte stream through its standard input and appends that data to a designated log file.

The process expects at least one command-line argument: the primary log file path. A second, optional argument may be supplied as a fallback path. If the primary log file cannot be opened or created, the process attempts to use the secondary path instead. This gives the system a simple recovery path when the preferred log location is unavailable.

The log file is opened lazily. In other words, the file is not opened during startup, but only when the first actual block of log data is received. This avoids creating empty important-log files in runs where no cortical or fatal messages were emitted.

During runtime, the process continuously reads from its inherited standard input handle, which is expected to be a pipe connected to the host application. Each successfully read chunk is appended to the selected output file. The process continues running until the pipe reaches end-of-file or breaks, which normally happens when the host closes the pipe during shutdown or when the host process terminates unexpectedly.

The implementation is deliberately simple because this code exists on the failure path. Complexity here would be counterproductive. The fewer dependencies and moving parts involved, the lower the chance that the emergency logging path fails for the same reason as the main process.

Host Application Integration

The host application must launch ImpLogProcess as a background child process and redirect a readable pipe into the child process standard input. The host keeps the write end of that pipe and uses it as a dedicated transport channel for important log records. Whenever a message must survive independently of the main logging path, the host writes the fully formatted bytes into that pipe.

This is the core integration model: the host selects and formats the important messages, while ImpLogProcess only persists them. The helper is therefore not a separate logging framework. It is an external append-only sink for already prepared critical log output.

A correct integration requires three things. First, the host must provide the log file path arguments. Second, it must ensure that the child process inherits a valid standard input handle. Third, it must keep the write end of the pipe open for as long as important log traffic may still be sent.

Startup Contract

At process startup, ImpLogProcess validates its required inputs immediately. If the primary log path argument is missing, null, or empty, the process exits with an error. It also verifies that standard input is valid. If standard input is not redirected correctly, startup fails because the process has no input channel from which to receive log data.

The host application must therefore prepare the child process correctly before launch. Simply starting the executable is not sufficient. The host must create the pipe, configure handle inheritance properly, bind the pipe's read end to the child standard input, and pass the intended file paths as arguments.

The file path arguments should refer to valid locations whose parent directories already exist. The helper can create the log file itself when needed, but it does not create missing directory trees.

Pipe and Process Wiring

The normal integration approach is to create an anonymous pipe and pass its read end to the child process as STDIN. The host retains the write end and writes log records into it as needed.

The handle ownership must be clean. The child should inherit only the read side of the pipe. The host should keep only the write side. After the child process has been created successfully, the host should close its own copy of the read handle. If this is not done correctly, end-of-file handling becomes unreliable because the system may still see an open writer or reader where it should not.

The child process is intended to run without user interaction and should typically be launched as a background helper without a visible window.

Data Contract

ImpLogProcess treats incoming data as an opaque byte stream. It does not know where one log record begins or ends unless the host itself defines those boundaries through what it writes. It also does not convert between encodings.

For that reason, the host application should send fully prepared log records. In practice, each important message should already contain everything needed for later analysis, such as timestamp, severity, source, and message text. Each record should normally end with a newline so that the resulting file remains readable.

A consistent encoding should be used for all writes. UTF-8 text is usually the most practical choice, but the helper will store whatever bytes it receives.

Log Selection in the Host

The host application should use this process only for high-value log traffic. Sending all normal diagnostic output through this channel would weaken the design by increasing load and turning a focused reliability mechanism into a second full logging system.

The intended use is to mirror only those messages that remain important when the application is already failing. Typical examples are fatal errors, crash-adjacent state messages, and other severe diagnostics that explain why the process is about to terminate or has entered an unrecoverable state.

This keeps the separate important-log file short, relevant, and useful during post-failure analysis.

Runtime Behavior

Once running, ImpLogProcess stays in a simple read-append loop. It reads chunks from standard input into a fixed-size buffer and writes those bytes to the log file. If the file has not yet been opened, it attempts to open the primary path first and, if that fails, the secondary path if one was provided.

Each chunk is written in full before the process continues to the next read. The implementation explicitly handles partial writes by retrying until the complete chunk has been appended. It also guards against zero-progress write situations by retrying only a limited number of times instead of risking an endless loop.

This behavior keeps the process deterministic and avoids silently discarding partially written data.

Shutdown Behavior

Normal shutdown is driven by the pipe. When the host application has no more important logs to send, it closes the pipe's write handle. The helper process then sees end-of-file on its standard input, exits the read loop, flushes the file buffers if the log file had been opened, closes the handle, and terminates successfully.

Unexpected host termination is handled through the same mechanism. If the host crashes, the operating system closes the host-owned pipe handles during process teardown. From the helper's point of view this appears as a broken pipe or end-of-file, which the implementation treats as a normal termination condition rather than an error. This is exactly the behavior required for crash-resilient logging.

Error Handling

The process uses explicit non-zero return codes to indicate failure conditions. These codes make integration and deployment problems easier to diagnose. Startup validation failures, read failures, file open failures, and write failures are distinguished so that the host or developer can identify where the failure occurred.

The current implementation uses explicit numeric exit codes. Those codes are useful during integration testing, diagnostics, and support:

  • 1: missing or invalid primary log file path argument
  • 2: missing or invalid standard input handle
  • 3: unexpected read failure
  • 4: failed to open both primary and secondary paths
  • 5: failed to open the primary path when no secondary path was provided
  • 6: file write failure
  • 7: repeated zero-byte write progress failure

These codes are mainly diagnostic. The host application does not need complex runtime handling around them, but it should at least detect whether the helper started successfully and whether it remained alive long enough to serve its purpose.

Recommended Host Responsibilities

The host application should treat ImpLogProcess as a best-effort reliability component and integrate it accordingly. It should start the helper early enough that important logs can be forwarded as soon as needed. It should format important records completely before writing them. It should keep the pipe handle valid for the full application lifetime during which such messages may occur. It should also close the pipe cleanly during normal shutdown so that the helper can flush and exit in an orderly way.

If desired, the host may also monitor the child process handle for unexpected early termination, especially during development or testing. That can help catch permission issues, invalid paths, or process creation mistakes.

Limitations

ImpLogProcess is intentionally not a full logging backend. It does not rotate files, manage retention, compress old logs, create directory trees, enrich records, or coordinate multiple log destinations. It writes only to a single chosen file target, with an optional fallback path at open time.

It is also not designed for heavy continuous traffic. The intended workload is infrequent, high-severity, high-value log output. That is the scenario for which the design is strongest and simplest.

Summary

ImpLogProcess is an external emergency log writer for critical events. The host application launches it, connects a pipe to its standard input, and forwards only fully formatted important log records. The helper process appends those bytes to a dedicated file, independently of the main application's internal logging health. This separation gives the system a reliable last-resort path for preserving the diagnostics that matter most when the main process is already failing.

#define WIN32_LEAN_AND_MEAN
#define NOMINMAX
#include <windows.h>
int wmain(int argc, wchar_t* argv[])
{
if(argc < 2 || argv[1] == nullptr || argv[1][0] == L'\0')
return 1; // Missing/invalid primary log file path argument
const wchar_t* priPath = argv[1];
const wchar_t* secPath = (argc >= 3 && argv[2] != nullptr && argv[2][0] != L'\0') ? argv[2] : nullptr;
const HANDLE hStdIn = ::GetStdHandle(STD_INPUT_HANDLE);
if(hStdIn == INVALID_HANDLE_VALUE || hStdIn == nullptr)
return 2; // Missing/invalid STDIN handle (pipe not redirected/inherited)
HANDLE hLogFile = INVALID_HANDLE_VALUE;
constexpr DWORD bufSize = 64u * 1024u; // 64 KiB
char buffer[bufSize]{};
while(true) {
DWORD bytesRead = 0;
const BOOL readSuccess = ::ReadFile(hStdIn, buffer, bufSize, &bytesRead, nullptr);
if(!readSuccess) {
const DWORD e = ::GetLastError();
if(e == ERROR_BROKEN_PIPE || e == ERROR_HANDLE_EOF)
break; // Normal termination: parent closed/crashed and pipe ended.
if(hLogFile != INVALID_HANDLE_VALUE)
::CloseHandle(hLogFile);
return 3; // Read failed (unexpected error)
}
if(bytesRead == 0)
break; // EOF (End of File) i.e. Main application closed
if(hLogFile == INVALID_HANDLE_VALUE) {
hLogFile = ::CreateFileW(priPath, FILE_APPEND_DATA, FILE_SHARE_READ, nullptr, OPEN_ALWAYS, FILE_ATTRIBUTE_NORMAL, nullptr);
if(hLogFile == INVALID_HANDLE_VALUE && secPath != nullptr) {
hLogFile = ::CreateFileW(secPath, FILE_APPEND_DATA, FILE_SHARE_READ, nullptr, OPEN_ALWAYS, FILE_ATTRIBUTE_NORMAL, nullptr);
if(hLogFile == INVALID_HANDLE_VALUE)
return 4; // Failed to open/create both primary and backup
}
else if(hLogFile == INVALID_HANDLE_VALUE) {
return 5; // Failed to open/create primary (no backup provided)
}
}
DWORD remains = bytesRead;
DWORD offset = 0;
int noProgressTries = 0;
while(remains > 0) {
DWORD written = 0;
const BOOL writeSuccess = ::WriteFile(hLogFile, buffer + offset, remains, &written, nullptr);
if(!writeSuccess) {
::CloseHandle(hLogFile);
return 6; // Write failed
}
if(written == 0) {
// No progress. Retry a few times, then give up.
if(++noProgressTries >= 16) {
::CloseHandle(hLogFile);
return 7; // No progress after many tries
}
::SwitchToThread(); // brief yield
continue;
}
// Progress
noProgressTries = 0;
offset += written;
remains -= written;
}
}
if(hLogFile != INVALID_HANDLE_VALUE) {
::FlushFileBuffers(hLogFile);
::CloseHandle(hLogFile);
}
return 0; // Normal shutdown
}
