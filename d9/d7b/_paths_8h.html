Detailed Description

Public API for file/folder paths used in the application.

This header declares functions that return various file and folder paths used in the application. The paths are primarily in two locations:

  • C:\ProgramData For application's internal data (e.g., settings, temp files).
  • C:\Users\Public\Documents For user-accessible files (e.g., logs, user data).
Note
  • The names and other meta-data related to the organization and application (e.g. application name, organization name, organization domain, etc.) are defined in Iden.hpp.
  • The system folder C:\ProgramData or C:\Users\Public\Documents is NOT created if it is missing; return empty instead.
  • Since the path is cached on first use, any changes after the first call are not visible.
The functions are organized into namespaces for clarity:
  • npi::dirs For directories.
  • npi::files For files.
Note
  • Functions in npi::dirs ensure the directories exist, creating them if necessary.
  • Functions in npi::files return file paths without creating the files.
Each function performs its work once, caches the absolute path in a function-local static, and returns the cached value on subsequent calls. If resolution or creation fails, an empty QString is returned.

The corresponding implementations (in Paths.cpp) prefer the Windows Known Folders API (SHGetKnownFolderPath for FOLDERID_ProgramData / FOLDERID_PublicDocuments) and fall back to environment variables when needed.

Note
  • These APIs are Windows-only. On non-Windows builds they return an empty QString.
  • No UI, no termination: call sites decide how to handle empty results (e.g., fail-fast at startup).
Warning
If a function returns an empty QString, it indicates a failure to resolve or create the requested path. Callers MUST check for empty results and handle them appropriately. The preferred approach is to treat it as a fatal error.

Usage Examples

int main(int argc, char** argv) {
QApplication app(argc, argv);
const QString progData = npi::dirs::programData();
const QString pubDocs = npi::dirs::publicDocuments();
if (progData.isEmpty() || pubDocs.isEmpty()) {
qCritical() << "Cannot resolve required data paths.";
return EXIT_FAILURE;
}
// Normal startup continues...
return app.exec();
}

Error Handling

  • No exceptions are thrown. An empty QString indicates failure.
  • Callers MUST check for empty and treat as a fatal startup error.

Thread Safety

Function-local static initialization is thread-safe in C++11 and later. After the first successful (or failed) resolution, subsequent calls are O(1).

Testing Tips

The resolved paths are cached per-process. If you need to test failure/success branches with different environments, run the tests in separate processes or abstract the resolution behind a seam. For simple tests, you may rely on the environment-variable fallbacks by ensuring the Known-Folder calls fail (e.g., via a test build flag) and using qputenv().
